Q1.A security engineer is troubleshooting an AWS Lambda function named MyLambdaFunction. The function encounters an error when it attempts to read objects in an Amazon S3 bucket named DOC-EXAMPLE-BUCKET. The S3 bucket has the following bucket policy:

{ "Effect": "Allow", "Principal": { "Service": "lambda.amazonaws.com" }, "Action": "s3:GetObject", "Resource": "arn:aws:s3:::DOC-EXAMPLE-BUCKET", "Condition": { "ArnLike": { "aws:SourceArn": "arn:aws:lambda:::function:MyLambdaFunction" } } }
Which change should the security engineer make to the policy to ensure that the Lambda function can read the bucket objects?
ARemove the Condition element. Change the Principal element to: {"AWS": "arn:aws:lambda:::function:MyLambdaFunction"}
BChange the Action element to: ["s3:GetObject*", "s3:GetBucket*"]
CChange the Resource element to "arn:aws:s3:::DOC-EXAMPLE-BUCKET/*"
DChange the Resource element to "arn:aws:lambda:::function:MyLambdaFunction". Change the Principal element to: {"Service": "s3.amazonaws.com"}
✓ Correct: C. Change the Resource element to "arn:aws:s3:::DOC-EXAMPLE-BUCKET/*".

How to Think About This:
When an S3 policy isn't working, always check that the Resource ARN scope matches the Action scope. S3 has two levels: bucket-level operations (like s3:ListBucket) use the bucket ARN, while object-level operations (like s3:GetObject) require the bucket ARN with a wildcard suffix (/*). This is one of the most common S3 policy mistakes on the exam.

Key Concepts:
S3 Bucket ARN vs Object ARN — arn:aws:s3:::DOC-EXAMPLE-BUCKET refers to the bucket itself (used for bucket-level actions like ListBucket, GetBucketLocation). arn:aws:s3:::DOC-EXAMPLE-BUCKET/* refers to all objects inside the bucket (used for object-level actions like GetObject, PutObject, DeleteObject).

S3 Bucket Policy Principal — The "Service": "lambda.amazonaws.com" principal with an ArnLike condition is a valid pattern for allowing a specific Lambda function to access the bucket via a resource-based policy. This part of the policy is correct.

Why C is correct: The action s3:GetObject operates on objects, but the Resource ARN arn:aws:s3:::DOC-EXAMPLE-BUCKET points to the bucket itself — not its contents. Adding /* changes the scope to all objects within the bucket, which is exactly what s3:GetObject needs to function. This is the only change required.

Why others are wrong:
A) Remove the Condition and set Principal to {"AWS": "arn:aws:lambda:::function:MyLambdaFunction"} — Lambda functions are not IAM principals. You cannot place a Lambda function ARN in the "AWS" principal field. Access from Lambda to S3 is granted either through the function's execution role (IAM role) or through a bucket policy using the "Service": "lambda.amazonaws.com" principal with a condition. The existing Principal/Condition pattern is already valid — the Resource is the problem.
B) Change the Action to ["s3:GetObject*", "s3:GetBucket*"] — Adding more actions does not fix the underlying resource scope mismatch. Even with additional actions, the Resource ARN still points to the bucket, not its objects. The original s3:GetObject action is sufficient — the action is not the problem.
D) Change Resource to Lambda ARN and Principal to s3.amazonaws.com — This is backwards. In an S3 bucket policy, the Resource must be an S3 ARN (the bucket or its objects), not a Lambda ARN. Setting the Principal to s3.amazonaws.com would mean the S3 service itself is the caller, which makes no sense for a Lambda-to-S3 access scenario.
